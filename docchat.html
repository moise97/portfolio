01 — The Problem

Document processing at Pfizer was still manual

During my AI externship at Pfizer, I was exposed to a recurring pain point across enterprise environments: critical information locked inside PDFs. Supply chain documents, clinical reports, compliance records — all sitting in static files that required someone to manually read through them to extract answers.

The question I was handed: can we build a system that lets a non-technical user ask questions about a document and get accurate, sourced answers in seconds?

The constraints were real. The solution had to handle both digital PDFs and scanned documents. It had to cite its sources so users could verify answers. And it had to be deployable — not just a notebook proof of concept.

02 — The Approach

Why RAG — and why not fine-tuning

The first architectural decision was the most important: RAG (Retrieval-Augmented Generation) versus fine-tuning a model on the documents.

Fine-tuning was ruled out quickly. It requires large amounts of labeled data, is expensive to retrain when documents update, and doesn't give you source citations out of the box. For a document intelligence use case where accuracy and traceability matter, RAG was the right call.

RAG works in two phases: retrieval (find the relevant chunks of the document) and generation (use an LLM to synthesize a natural language answer from those chunks). The key variable is the embedding model — how you convert text into vectors that can be compared for semantic similarity.

🔬
Embedding model comparison

I tested three embedding models against the same document set: MiniLM (fast, lightweight), E5 (strong multilingual performance), and BGE (best retrieval accuracy on technical documents). BGE won on precision — it consistently returned the most relevant chunks for domain-specific queries.

For the retrieval layer I used LlamaIndex to manage document ingestion, chunking, and vector indexing. For the generation layer, Gemini API handled response synthesis. The initial version used TF-IDF (term frequency-inverse document frequency) for a fast, dependency-light baseline before moving to dense embeddings.

03 — What Broke

Three problems I didn't anticipate

1

Dependency conflict between transformers and Gradio

When I tried to add dense embedding support, the transformers library version required by the embedding models conflicted with the Gradio version required for the UI. The app would import fine but crash on launch with a silent error. Took several hours of isolation testing to identify — the fix was pinning specific versions in requirements.txt and restructuring the import order.

2

Scanned PDFs returned empty text

pdfplumber extracts text from digital PDFs perfectly. But scanned PDFs — images of pages — have no extractable text layer. The first version of DocChat silently returned nothing for these files, which was worse than an error. The fix was an OCR fallback: detect when extraction returns empty, then pipe the page images through pytesseract to extract text before indexing.

3

Gemini API rate limiting under load

During testing with multiple document queries in quick succession, the Gemini API started returning 429 errors. The naive fix (retry immediately) made it worse. The real fix was exponential backoff — wait 1s, then 2s, then 4s between retries — combined with request queuing so the UI never hung silently.

04 — What I Decided

Architecture decisions that made it production-ready

After working through the problems above, the final architecture looked like this:

1
Upload
User uploads PDF via Gradio two-panel UI
→
2
Extract
pdfplumber extracts text; pytesseract OCR fallback for scanned pages
→
3
Index
LlamaIndex chunks and indexes with BGE embeddings
→
4
Retrieve
Top-k semantic search returns relevant chunks
→
5
Generate
Gemini API synthesizes answer with source citations

The two-panel Gradio UI keeps document context on the left, conversation on the right. Every answer includes the source chunk and page number so users can verify. This was a deliberate design decision — in a regulated environment like pharma, answer traceability isn't optional.

💡
The surgical parallel

In an operating room, every instrument is accounted for before and after a procedure. You don't assume — you verify. I applied the same logic to RAG: every answer the system gives must be traceable back to a source. If it can't cite it, it shouldn't say it.

05 — Outcome

Deployed, live, and still running

DocChat is deployed live on HuggingFace Spaces and handles both digital and scanned PDFs. Users can upload any document and ask questions in natural language — the system returns answers with source citations in seconds.

The project demonstrated that a production RAG pipeline doesn't require a massive engineering team or enterprise infrastructure. A well-architected Python stack — LlamaIndex, Gemini, Gradio, pytesseract — deployed on HuggingFace is enough to solve a real problem for real users.

✓ Handles digital + scanned PDFs via OCR fallback
✓ Source citations on every answer for traceability
✓ Rate limit handling with exponential backoff
✓ BGE embeddings outperform MiniLM and E5 on technical docs
✓ Deployed live — not a notebook, a real product

DocChat Pro (in progress) extends this with advanced chunking strategies, multi-document support, and a more refined retrieval configuration. The foundation is solid — the next version builds on what this one proved.

06 — What I Learned

The unglamorous truth about shipping AI

The hardest parts of building DocChat had nothing to do with AI. They were dependency management, silent failure modes, and rate limiting. The model was the easy part. The engineering around the model was where the real work happened.

That's the gap between a demo and a product. Anyone can get a RAG pipeline returning answers in a Colab notebook. Making it handle edge cases gracefully, fail loudly instead of silently, and stay up under real usage — that's engineering.

I came away with a much clearer mental model of what "production-ready" actually means: not just working, but failing predictably, recovering automatically, and communicating clearly when it can't.
